# Adaptive Pace — LLM 목소리 재현 테스트 검증 계획

> MVP 배포 전 LLM 재가공 품질을 수백 회 반복 검증하기 위한 체계적 테스트 프레임워크

---

## 1. 테스트 목표

| 목표 | 기준 |
|------|------|
| 핵심 메시지 유지율 | 원본과 재가공본의 핵심 주장이 동일해야 함 (목표: 95%+) |
| 구조 변형률 | 원본과 텍스트 유사도 30% 이하 (= 70% 이상 다른 텍스트) |
| 톤/목소리 일치도 | 원본 작성자가 "내가 쓴 것 같다"고 느끼는 비율 (목표: 75%+) |
| 플랫폼 적합성 | LinkedIn/Threads/Bluesky 각 포맷에 맞는 길이·구조 |
| 환각(Hallucination) 발생률 | 원본에 없는 사실/수치를 지어내는 비율 (목표: 0%) |

## 2. 테스트 데이터셋 구성

### 2.1 크리에이터 페르소나 (최소 5종)

| # | 페르소나 | 톤 특성 | 언어 |
|---|---------|---------|------|
| P1 | 스타트업 마케터 (직설적, 짧은 문장) | "솔직히", 끊어치기, 이모지 적음 | 한국어 |
| P2 | 테크 블로거 (분석적, 데이터 중심) | 수치 인용, 비교표, 길고 논리적 | 영어 |
| P3 | 라이프 코치 (감성적, 공감형) | 질문형 훅, 스토리텔링, 이모지 많음 | 영어 |
| P4 | B2B SaaS 창업자 (실전형, Build in Public) | 매출 공개, 실패담, 짧고 임팩트 | 영어 |
| P5 | 프리랜서 디자이너 (캐주얼, 유머) | 밈 레퍼런스, 비격식, 줄임말 | 영어 |

### 2.2 페르소나당 원본 글 구성

- **최소 10개** 원본 글 수집/작성
- 다양한 구조 포함: 리스트형, 스토리텔링, 질문형, 데이터형, 짧은 글(3줄), 긴 글(20줄+)
- 총 테스트 데이터셋: **5 페르소나 × 10개 = 50개 원본**

### 2.3 데이터 폴더 구조

```
tests/
├── voice-test/
│   ├── personas/
│   │   ├── p1-startup-marketer.json    ← 원본 글 10개 + 스타일 프로필
│   │   ├── p2-tech-blogger.json
│   │   ├── p3-life-coach.json
│   │   ├── p4-saas-founder.json
│   │   └── p5-freelance-designer.json
│   ├── results/
│   │   └── {timestamp}-{model}-{persona}.json  ← 매 실행 결과 저장
│   ├── run-voice-test.js               ← 자동화 테스트 스크립트
│   └── analyze-results.js              ← 결과 분석/리포트 생성
```

## 3. 테스트 시나리오

### 3.1 기본 재가공 테스트

```
입력: 원본 글 1개 + 같은 페르소나의 다른 글 4개 (few-shot)
출력: 재가공된 글
검증: 메시지 유지, 구조 변형, 톤 일치
```

### 3.2 모델 비교 테스트

동일 입력을 여러 모델로 돌려서 품질 비교:

| 모델 | 용도 |
|------|------|
| gemini-2.5-flash | 메인 후보 (비용/품질 밸런스) |
| gemini-2.5-flash-lite | 저비용 후보 |
| gemini-2.5-pro | 품질 상한선 확인용 |
| gpt-4o-mini | 크로스 벤더 비교 (선택) |

### 3.3 few-shot 개수 테스트

스타일 참고 글 개수에 따른 품질 변화 측정:

| 테스트 | few-shot 개수 | 목적 |
|--------|-------------|------|
| A | 0개 (제로샷) | 베이스라인 |
| B | 2개 | 최소 참고 |
| C | 5개 | 기본 설정 |
| D | 10개 | 최대 참고 |

### 3.4 프롬프트 변형 테스트

| 변형 | 프롬프트 차이점 |
|------|---------------|
| V1 (기본) | "이 사람의 스타일로 재작성하세요" |
| V2 (구체적 지시) | 문장 길이/이모지/훅 스타일을 명시적으로 지정 |
| V3 (네거티브) | "설명을 늘리지 마세요. 문장을 길게 만들지 마세요" 추가 |
| V4 (스타일 분석 단계 분리) | 1단계: 스타일 분석 출력 → 2단계: 재가공 (Chain of Thought) |
| V5 (시스템 프롬프트 활용) | 시스템 프롬프트에 페르소나 고정 |

### 3.5 플랫폼 전환 테스트

같은 원본을 다른 플랫폼 포맷으로 변환:

| 원본 플랫폼 | 타겟 플랫폼 | 검증 포인트 |
|------------|-----------|------------|
| LinkedIn | Threads | 길이 축소 (500자→280자), 톤 캐주얼화 |
| LinkedIn | Bluesky | 300자 제한, 쓰레드 분할 |
| Threads | LinkedIn | 길이 확장, 톤 프로페셔널화 |

### 3.6 엣지 케이스 테스트

| 케이스 | 입력 | 기대 동작 |
|--------|------|----------|
| 극히 짧은 원본 | "오늘 정말 힘든 하루였다." (한 줄) | 억지로 늘리지 않고, 짧게 변형 |
| 매우 긴 원본 | 2000자+ 에세이 | 핵심만 추출하여 플랫폼에 맞게 축약 |
| 데이터/숫자 포함 | "매출 $50K, 전환율 3.2%" | 숫자를 변조하지 않아야 함 (환각 테스트) |
| 링크 포함 | URL이 들어간 글 | 링크를 유지하거나 적절히 처리 |
| 이모지 헤비 | 이모지가 10개+ 인 글 | 이모지 스타일 유지 |
| 민감한 주제 | 실패담, 해고 경험 등 | 톤을 바꾸되 메시지를 왜곡하지 않음 |

## 4. 자동화 평가 지표

### 4.1 자동 측정 가능 지표

| 지표 | 측정 방법 | 합격 기준 |
|------|----------|----------|
| **텍스트 유사도** | cosine similarity (임베딩 비교) | 원본 대비 0.3~0.7 (너무 같아도, 너무 달라도 실패) |
| **길이 비율** | 재가공 글자수 / 원본 글자수 | 0.7 ~ 1.5 (±50% 이내) |
| **이모지 일치** | 원본 이모지 사용량 대비 차이 | ±30% 이내 |
| **문장 평균 길이** | 재가공 평균 문장 길이 / 원본 평균 | ±40% 이내 |
| **환각 체크** | 원본에 없는 고유명사/숫자 감지 | 0건 |
| **금지어 체크** | "AI가 작성", "재가공", "원본" 등 메타 텍스트 | 0건 |

### 4.2 LLM-as-Judge (자동 정성 평가)

별도 LLM을 심판으로 사용하여 정성 평가 자동화:

```
프롬프트:
"아래 [원본]과 [재가공]을 비교하여 1~10점으로 평가하세요.

평가 기준:
1. 핵심 메시지 보존 (1~10)
2. 톤/목소리 일치도 (1~10)
3. 구조 변형 정도 (1~10, 너무 같으면 낮은 점수)
4. 자연스러움 (1~10, 사람이 쓴 것처럼 느껴지는지)
5. 플랫폼 적합성 (1~10)

JSON으로 출력: {message: N, tone: N, structure: N, natural: N, platform: N}"
```

### 4.3 합격 기준 종합

| 등급 | 평균 점수 | 판정 |
|------|----------|------|
| A | 8.0+ | 배포 가능 |
| B | 7.0~7.9 | 프롬프트 튜닝 후 재테스트 |
| C | 6.0~6.9 | 모델 업그레이드 또는 아키텍처 변경 검토 |
| F | 6.0 미만 | 배포 불가, 근본적 재설계 필요 |

## 5. 테스트 실행 프로토콜

### 5.1 1차 테스트 (프롬프트 탐색)

```
목적: 최적 프롬프트 조합 찾기
규모: 5 페르소나 × 5 프롬프트 변형 × 3 모델 = 75회
기간: 1일
산출물: 최적 프롬프트 + 최적 모델 조합
```

### 5.2 2차 테스트 (대량 검증)

```
목적: 최적 조합의 안정성 검증
규모: 5 페르소나 × 10 원본 × 3회 반복 = 150회
기간: 1일
산출물: 평균 점수, 분산, 최악 케이스 분석
```

### 5.3 3차 테스트 (엣지 케이스)

```
목적: 극단적 입력에서도 안전한지 확인
규모: 6 엣지 케이스 × 5 페르소나 × 3회 = 90회
기간: 0.5일
산출물: 엣지 케이스별 통과/실패 리포트
```

### 5.4 최종 테스트 (배포 전 회귀)

```
목적: 코드 변경 후 품질 퇴보 없는지 확인
규모: 핵심 50개 케이스 × 1회 = 50회
기간: 자동화 (CI에서 실행)
산출물: PASS/FAIL 판정
```

**총 테스트: 약 365회+**

## 6. 결과 리포트 형식

매 테스트 실행 후 자동 생성:

```
=== Voice Replication Test Report ===
Date: 2026-XX-XX
Model: gemini-2.5-flash
Prompt: V3 (네거티브 지시 포함)
Few-shot: 5개

총 테스트: 50건
합격(A+B): 42건 (84%)
불합격(C+F): 8건 (16%)

평균 점수:
  메시지 보존: 8.7/10
  톤 일치도:  7.2/10  ← 개선 필요
  구조 변형:  7.8/10
  자연스러움: 7.5/10
  플랫폼 적합: 8.1/10
  종합: 7.86/10 (등급: B)

환각 발생: 1건 (글 #P2-07, 없는 통계 수치 생성)
금지어 발생: 0건

최악 케이스:
  P5-03: tone=4.0 (캐주얼 톤이 포멀하게 변환됨)
  P1-08: structure=5.0 (원본과 거의 동일한 구조)

권장 액션:
  1. V3 프롬프트에 "캐주얼 톤 유지" 지시 강화
  2. 짧은 원본(3줄 이하)에 대한 별도 프롬프트 분기
```

## 7. 알려진 이슈 및 튜닝 포인트

### 초기 테스트에서 발견된 문제 (2026-04-08)

| 문제 | 원인 | 해결 방향 |
|------|------|----------|
| LLM이 문장을 길게 늘리는 경향 | 모델의 기본 성향 (verbose) | V3 프롬프트: "문장을 짧게 유지. 설명 추가 금지" |
| 원본보다 정중해지는 현상 | 모델의 safety alignment | "원본의 캐주얼 수준을 유지하세요" 명시 |
| 구조가 원본과 너무 유사 | few-shot이 구조까지 고정시킴 | "구조를 반드시 변경하세요 (리스트→서술 또는 반대)" 강제 |
| gemini-2.5-flash 간헐적 503 에러 | 수요 과부하 | flash-lite 폴백 로직 필수 |
| gemini-2.0-flash 모델 폐지 | 2026년 3월 deprecated | 2.5 버전만 사용, 모델 버전 관리 필요 |
