# Adaptive Pace — 의사결정 로그

> 이 문서는 Adaptive Pace가 CONDITIONAL GO 판정을 받기까지의 맥락과 의사결정 과정을 기록한다.

---

## 1. 파이프라인 여정

### Round 3 전체 흐름

Round 3는 Phase 1에서 "프로토콜 레벨 미들웨어"와 "규제 준수 도구"를 중심으로 강한 GO 후보 4개를 확보했다:

| # | GO 후보 | 최종 결과 |
|---|---------|----------|
| #1 | LinkedIn HMAC Proxy Guard | **탈락** — S1(Zapier/Make 네이티브 처리), SOM 69~690명 |
| #2 | RFC 8058 Unsubscribe Endpoint | **탈락** — SES ListManagementOptions 네이티브 해결 |
| #3 | EU AI Act GPAI Renderer | **CONDITIONAL GO** |
| #12 | Ghost CMS Pagination Proxy | **탈락** — SOM 500~3,000명, 10줄 코드 자체 해결 |

**GO 후보 4개 중 3개가 탈락.** 공통 패턴: "전용 경쟁자 0개"가 기회가 아니라 **수요 부재 시그널**(L1 "블루오션에 물고기 없음")이었다.

### MAYBE에서 GO로 — #9의 반전

Adaptive Pace(#9)는 Gate 1 스크리닝에서 **MAYBE**로 분류되었다 (H2 WARNING + H4 WARNING). 소셜 스케줄링 시장이 극도로 포화되어 있어 GO 후보보다 낮은 등급을 받았다.

그러나 LR5 교훈("MAYBE 후보를 건너뛰지 마라 — Round 2에서 Neolemon이 MAYBE에서 유일한 CONDITIONAL GO가 된 선례")을 적용하여 풀 토론을 진행한 결과:

- Gate 2: WARNING 2개 (S1, S3) — 고위험 플래그 미달
- Gate 3: 3라운드 토론 → **찬성파 5승, 무승부 1개, 반대파 0승** → CONDITIONAL GO

**Round 2 Neolemon과 동일한 패턴: GO 후보가 전멸한 후 MAYBE에서 실질적 최선 후보가 발견됨.**

## 2. #1/#2와 #9의 구조적 차이: "역인센티브 니치"

### #1 LinkedIn HMAC, #2 RFC 8058의 L1 패턴

#1과 #2는 "변화 갭 + 기술"이라는 강력한 조합이었지만, **플랫폼이 이미 자체 해결**한 문제였다:

- #1: Zapier/Make 네이티브 LinkedIn Ads 앱이 HMAC을 자동 처리 → 핵심 가치 소멸
- #2: Amazon SES v2 `ListManagementOptions`가 RFC 8058을 네이티브 해결 → 핵심 가치 소멸

**교훈**: "전용 경쟁자 0개"는 기회일 수도 있고, 플랫폼이 이미 해결했기 때문에 수요가 없는 것일 수도 있다.

### #9의 "역인센티브 니치" = 플랫폼이 만들 수 없는 기능

Adaptive Pace의 핵심 차별화인 "번아웃 감지 → 사용자에게 쉬라고 권유"는 VC 기업이 구조적으로 만들 수 없다:

| VC 기업의 KPI | "쉬라고 하는 기능"의 효과 | 결과 |
|--------------|----------------------|------|
| MAU 증가 | MAU 감소 | **충돌** |
| 사용 시간 증가 | 사용 시간 감소 | **충돌** |
| 포스트 수 증가 | 포스트 수 감소 | **충돌** |

Buffer가 Pause를 "mental health toggle"이라 부르면서도 적극 push하지 않는 이유가 이것이다.

**이것이 #1/#2(플랫폼이 해결 가능)와 #9(플랫폼이 해결 불가)의 핵심 차이.**

## 3. #3 EU AI Act와의 비교

Round 3에서 CONDITIONAL GO를 받은 후보는 #3과 #9 두 개다.

| 비교 항목 | #3 EU AI Act GPAI Renderer | #9 Adaptive Pace |
|----------|--------------------------|-----------------|
| 해자 | 규제 의무 + 셀프서브 부재 | 역인센티브 니치 + 에버그린 재가공 |
| WTP 근거 | 벌금 최대 EUR35M (의무적) | OneUpApp $83K~$100K MRR (검증됨) |
| SOM | GPAI 모델 개발 SME 수백~수천 | 번아웃 경험 크리에이터 5,000~20,000 |
| 법적 리스크 | **L38 존재** — 규제 산업 1인 개발 구조적 불리. EUR35M 벌금 맥락에서 1인 도구 신뢰 장벽 | **없음** — 법적 리스크 해당 없음 |
| 시간 압박 | 2026.08.02 전 MVP 출시 필수 | 시간 압박 없음 |
| 스케일 천장 | $750 MRR 현실적이나 그 이상 불확실 | $750 MRR 현실적, OneUpApp 선례로 상향도 가능 |
| 타겟 | 엔지니어 (셀프서브 친화) | 크리에이터/스몰비즈 (셀프서브 친화) |

**#3의 강점**: 법적 의무가 WTP를 구조적으로 보장. "선택이 아닌 의무적 인프라."
**#3의 약점**: L38 심리적 신뢰 장벽, 시간 압박, SOM 천장.
**#9의 강점**: 법적 리스크 0, 시간 압박 없음, OneUpApp으로 WTP/스케일 검증.
**#9의 약점**: "번아웃 감지" 독립 WTP 미검증, MeetEdgar와 직접 경쟁.

## 4. 토론 핵심 공방 요약

### Gate 3 — 3라운드 진행

#### 라운드 1: 반대파 REFUTE vs 찬성파 REFUTE

| 공격 | 반박 | 승자 |
|------|------|------|
| H2: Buffer Pause Queue가 이미 mental health toggle 제공 | Buffer Pause = 수동 온/오프. 번아웃에 빠진 사람은 자신이 번아웃인지 모른다. Adaptive Pace = **자동 감지 + 자동 모드 전환** = "자동화/데이터 처리" 차별화 (T3: WARNING) | **찬성파** |
| H4: Buffer($20M ARR)/Sprout Social($360M ARR)이 추격 가능 | VC 메트릭(MAU/사용시간)과 "쉬라고 하는 기능"이 구조적 충돌. "역인센티브 니치" | **찬성파** |
| LC1: 번아웃 감지 = 사용을 줄이라는 역순환 | 순환의 핵심은 "에버그린 콘텐츠 자동 재가공". 재가공 콘텐츠 소비 → 새 배치 필요 = 콘텐츠 소비 순환. OneUpApp $83K MRR로 검증 | **찬성파** |
| L3: Buffer Free, Later Free, Canva Free로 무료 사용 가능 | 무료 도구는 스케줄링만 제공. 번아웃 감지 + 에버그린 재가공 + Active↔Survival 자동 전환 = 0개 제품이 제공 | **찬성파** |

#### 라운드 2: 반대파 PARTIAL (H4, LC1 수용) vs 찬성파 PARTIAL

| 공격 | 반박 | 승자 |
|------|------|------|
| "번아웃 감지"의 독립적 WTP 분리 불가. OneUpApp은 번아웃 감지 없이 달성 | 수용. 포지셔닝 재정의: "에버그린 재가공 도구 + 번아웃 감지 보너스". 핵심 WTP = 에버그린 재가공, 차별화 = 번아웃 감지, 마케팅 훅 = #CreatorBurnout | **찬성파** (정직한 포지셔닝 전환) |
| S3: $750 MRR 달성 경로 불확실 ($150~$1,250 편차) | $15/월 x 50명 = $750. ProductHunt + Reddit r/Entrepreneur 초기 트래픽. MeetEdgar 이탈 가격 민감 크리에이터 타겟. L9: 실패 비용 $0 | **무승부** |

#### 라운드 3: 반대파 PARTIAL (CONCEDE 근접) vs 찬성파 REFUTE

반대파가 핵심 논거(에버그린 재활용 WTP + 역인센티브 니치)를 수용하고, 잔여 우려(X API 비용, API 유지보수, Buffer Free 대비)만 남김. 찬성파가 MVP 스코프 조절(X 제외, 3개 플랫폼만)로 모든 실행 리스크를 해소.

### 심판 최종 판정

| 쟁점 | 승자 |
|------|------|
| H2 (무료 대안) | **찬성파** — 자동 감지+재가공 = T3 WARNING 정당 |
| H4 (VC 경쟁) | **찬성파** — 역인센티브 니치 |
| LC1 (내부 순환) | **찬성파** — 에버그린 재가공 순환 |
| S3 (SOM) | **무승부** — 가능하나 불확실 |
| L3 (무료 표준) | **찬성파** — 조합 기능 미제공 |
| X API 비용 | **찬성파** — MVP에서 X 제외 |

**종합: 찬성파 5승, 무승부 1개, 반대파 0승 → CONDITIONAL GO**

## 7. 의사결정 원칙 정리

이 프로젝트의 GO 판정에서 도출된 핵심 원칙:

1. **"전용 경쟁자 0개"를 무조건 기회로 보지 말 것** — #1, #2가 이 함정에 빠짐. 플랫폼 네이티브 해결 여부를 반드시 확인
2. **"역인센티브 니치"가 인디 개발자의 구조적 우위** — VC가 구조적으로 진입 못 하는 영역을 찾을 것
3. **MAYBE 후보를 건너뛰지 말 것 (LR5)** — Round 2 Neolemon, Round 3 Adaptive Pace 모두 MAYBE에서 최선 후보가 나옴
4. **핵심 WTP와 마케팅 훅을 분리할 것** — 에버그린 재가공이 WTP, 번아웃 감지가 마케팅 훅. 혼동하면 제품 설계 오류
5. **실패 비용이 $0이면 불확실해도 시도할 가치 있음 (L9)** — SOM 불확실성이 있으나 초기 비용이 거의 없으므로 검증 시도 합리적
